# Test coverage reports

**Date:** 2026-07-24
**Author:** Michael Biehl
**Status:** in-review
**Origin:** ai-assisted
**Human-validated:** false
**Review-PR:** https://github.com/crystldm/reinicorn-kb/pull/4

## Problem

The suite has 702 tests across `tests/`, and `.github/workflows/test.yml`
runs `ruff`, `pyright`, and `pytest` on every push and pull request. Nothing
measures how much of `src/reinicorn` those tests actually exercise.

Consequences today:

- No signal when a PR adds a module or branch with no test behind it. A
  reviewer has to notice by eye.
- No way to see which parts of the CLI are thin on coverage when deciding
  where to spend test effort.
- An ad-hoc measurement (`uv run --with pytest-cov pytest --cov=src/reinicorn`)
  puts the project at **85.69%** with branch coverage on (3436 statements,
  406 missed; 1176 branches, 184 partial). Statement-only coverage reports a
  flattering **86–87%** for the same suite, because it counts a half-taken
  `if` as fully covered. Either way the number is invisible unless someone
  runs it by hand, and it can silently drift.

The weakest areas are `commands/lint.py` (0%), `linter/runner.py` (45%), and
`commands/kb_manage.py` (76%). Those are findings, not part of this change —
this spec is about the measurement infrastructure, not about raising the
number.

## Design Goals

- `uv run pytest` produces a coverage report locally with no extra flags.
- CI fails when total line coverage drops below an agreed floor.
- A reviewer can see the coverage number and the per-file breakdown from the
  pull request without downloading anything.
- A browsable line-by-line HTML report is available from a CI run when a
  reviewer wants to dig in.
- Coverage data does not leave the repository. No third-party service, no
  upload token, no bot comment. This keeps forks and untrusted-PR runs
  working with zero secrets.
- Coverage measurement adds one dev dependency and no runtime dependency.
  Reinicorn ships with zero runtime dependencies and that does not change.

## Design

### 1. Dev dependency

Add `pytest-cov` to the `dev` dependency group in `pyproject.toml`. It pulls
in `coverage` transitively. No runtime dependency is added, and the sdist and
wheel build targets are untouched.

### 2. Coverage config in `pyproject.toml`

Configuration lives in `pyproject.toml` alongside the existing `ruff`,
`pyright`, and `pytest` config — no new dotfile.

`[tool.pytest.ini_options]` gains `addopts` so a bare `uv run pytest` is
already measured:

```
addopts = "--cov --cov-report=term-missing"
```

`[tool.coverage.run]` measures the installed package by import name rather
than by path, so the numbers are the same whether the suite runs against the
source tree or an installed wheel:

```
source = ["reinicorn"]
branch = true
```

Branch coverage is on. The CLI is dense with conditionals, and statement-only
coverage would call a half-taken `if` fully covered.

`[tool.coverage.report]` sets the floor and the exclusions:

```
fail_under = 85
show_missing = true
exclude_also = [
    "if TYPE_CHECKING:",
    "raise NotImplementedError",
    "if __name__ == .__main__.:",
]
```

`fail_under = 85` sits just under today's 85.69% branch-coverage total. It
catches a real regression without failing on the sub-1% noise of a normal PR.
It is a ratchet: raise it as coverage rises, never lower it to make a red
build green.

`src/reinicorn/_version.py` is generated by `hatch-vcs` and is already
excluded from `ruff`; it is added to `omit` here for the same reason.

### 3. CI reporting

`.github/workflows/test.yml` changes in the `Run tests` step and gains two
steps after it.

The test step emits three formats in one run — terminal for the log, XML for
the summary step, HTML for the artifact:

```
uv run pytest tests/ -v --cov-report=xml --cov-report=html
```

The step-summary step renders the per-file table into the GitHub Actions job
summary, so the number and the breakdown are visible from the PR's Checks tab
with no download. It runs `uv run coverage report --format=markdown` and
appends the output to `$GITHUB_STEP_SUMMARY`.

The artifact step uploads `htmlcov/` via `actions/upload-artifact`, giving a
browsable line-by-line report for anyone who wants to dig past the table.

Both new steps run with `if: always()` so a failing threshold still reports
where the gap is — the most useful moment to see the table is when the build
just went red.

`actions/upload-artifact` is pinned to a commit SHA with a version comment,
matching how `actions/checkout` and `astral-sh/setup-uv` are already pinned in
this workflow.

### 4. Ignored artifacts

`.coverage`, `coverage.xml`, and `htmlcov/` are generated on every local test
run. They are added to `.gitignore` under the existing "Test caches" heading.

### 5. Contributor documentation

`CONTRIBUTING.md` documents that `uv run pytest` reports coverage, states the
85% floor, and notes that CI publishes the table to the job summary. This is
the file that already tells contributors `pytest`, `ruff`, and `pyright` must
be green before review.

## Non-Goals

- **Raising coverage.** The gaps this surfaces (`linter/runner.py`,
  `commands/lint.py`) are followup work, not part of this change.
- **Codecov, Coveralls, or any external service.** Rejected deliberately:
  it would add a token, a bot comment, a third-party data path, and a
  fork-PR failure mode, in exchange for a badge.
- **Per-PR diff coverage** ("are the changed lines covered?"). That needs
  either a service or a base-branch comparison job. The total-with-floor is
  the cheap 80% of the value; diff coverage can come later if the floor
  proves too blunt.
- **A README coverage badge.** A badge needs a hosted number, which means a
  service. Out of scope by the same reasoning.
- **Per-file minimum coverage.** One global floor for now.
- **Coverage for `tests/` itself or for the shipped `hooks/` and `linters/`
  shell assets.** Only `src/reinicorn` is measured.
